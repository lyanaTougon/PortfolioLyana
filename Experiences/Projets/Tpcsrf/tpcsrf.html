🛡️ TP CSRF - Cross-Site Request Forgery

Objectif du TP

Comprendre et tester les attaques CSRF et les mécanismes de protection avec des tokens. L’objectif est de voir comment un site peut se protéger contre des requêtes malveillantes.

Étapes à suivre

  1. Lancer index.php et saisir un nom et un email.
  2. Soumettre le formulaire qui appelle submit.php et affiche les données.
  3. Tester une requête GET modifiée pour simuler une attaque CSRF :
    http://localhost/TP-CSRF/submit.php?fullname=Usurpateur&email=exec(mkfs.ext4 /dev/sdb1)&jeton=codeinconnu&submit=Submit
  4. Observer le message d’erreur si le jeton n’est pas valide.

Explications

Qu’est-ce que le CSRF ?

CSRF signifie falsification de requête inter-site. C’est une attaque où un site malveillant peut faire exécuter des actions sur un site légitime où tu es déjà connecté, sans ton consentement.

Exemple : un utilisateur connecté à son compte bancaire pourrait voir une requête externe déclencher un transfert d’argent automatiquement, car le navigateur envoie ses cookies de session.

Comment le TP protège contre CSRF

  • Chaque formulaire reçoit un token CSRF unique, lié à la session de l’utilisateur.
  • Le token est envoyé avec le formulaire et vérifié côté serveur à la soumission.
  • Si le token correspond à celui de la session → la requête est valide.
  • Si le token est absent ou incorrect → la requête est rejetée et un message d’alerte s’affiche.

Exemple d’attaque CSRF

Un attaquant pourrait essayer de soumettre une requête en dehors du formulaire original. Comme il ne connaît pas le token valide, la tentative est détectée et bloquée. Seul le site d’origine peut générer un token valide.

Importance de la session

Le token est stocké dans la session de l’utilisateur. Cela permet de lier le token à un utilisateur précis et d’empêcher qu’un token soit réutilisé par un attaquant. Sans session, le token serait le même pour tout le monde et la protection serait inefficace.
